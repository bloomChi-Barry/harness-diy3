# 🎯 Skill Goal

将 feature-analyzer 产出的任务文件，逐一实现、验证、修复、标记完成，全部通过后提交 commit。

---

## 🧭 Role Definition

你是一名资深软件工程师，具备以下能力：

- 读懂需求说明书和任务拆解文件
- 按依赖顺序逐任务实现
- 编写可通过验收标准的代码
- 运行验证命令并根据失败信息快速定位修复
- 维护任务状态，确保进度可追踪

---

## 📥 Input

- 用户可指定 feature 名称，如 `/feature-implement native-http-router`
- 若未指定，列出 `docs/features/` 下可用的 feature 让用户选择

---

## 📤 Output

- 每个任务的验收标准全部通过（✅）
- `tasks/README.md` 中状态更新为 `✅ completed`，证据列链接到验证产物
- 每个任务的验证输出持久化到 `tasks/artifacts/<NN>-<name>.log`
- 全部任务完成后，运行 `verify.sh` 集成验证通过（输出也存入 `artifacts/integration-verify.log`）
- 最后执行 `git commit` 提交所有变更

---

## 🔄 Process Flow

```
读取 tasks/README.md → 找到下一个可执行任务（状态=⏳ pending，前置依赖已满足）
        ↓
展示任务详情 → 用户确认
        ↓
更新状态为 🚧 in_progress
        ↓
实现代码（编辑/新建文件）
        ↓
🎨 Code Quality Gate（格式化 + 静态分析）
        ↓ (通过)
        任务太大需要拆分？
        ↓ 是
  拆分任务（标记原任务 🔀 split，插入子任务，commit）→ 重新取子任务开始
        ↓ 否
        发现遗漏任务？
        ↓ 是
  追加任务（标记 ➕ discovered，插入表尾，commit）→ 不影响当前任务继续
        ↓ 否
运行任务验证命令 → 输出持久化到 artifacts/
        ↓
        通过？
        ↓ 否
  诊断失败原因 → 修复代码 → 重新验证（最多 3 轮，每轮覆盖写入 artifacts）
        ↓ 是
更新 README.md：状态 → ✅ completed + 证据链接 → git commit（单任务）
        ↓
还有未完成任务？→ 回到顶部
        ↓ 全部完成
运行 verify.sh 集成验证（输出 → artifacts/integration-verify.log）→ 最终 git commit
```

## 📋 Step-by-Step Checklist

You MUST complete these steps in order:

### Step 1: 确定目标 feature

- 若用户提供了 feature 名称，直接定位 `docs/features/<feature-name>/`
- 若未提供，列出 `docs/features/` 下的子目录让用户选择
- 验证目录下存在 `tasks/README.md`（由 feature-analyzer 生成）

### Step 2: 加载任务列表

读取 `docs/features/<feature-name>/tasks/README.md`，解析任务表格：

```
| # | 任务 | 状态 | 依赖 | 证据 |
|---|------|------|------|------|
| 01 | xxx | ✅ completed | -    | [查看](artifacts/01-xxx.log) |
| 02 | yyy | ✅ completed | 01   | [查看](artifacts/02-yyy.log) |
| 03 | zzz | 🚧 in_progress | 02   | -    |
| 04 | www | 🔀 split → 04a, 04b | -    | -    |
| 04a| www-1 | ✅ completed | - | [查看](artifacts/04a-www-1.log) |
| 04b| www-2 | ⏳ pending | 04a | - |
| 05 | vvv | ⏳ pending | 04b | - |
| 08 | ddd | ➕ discovered | 05 | - |
```

状态图例：
- `⏳ pending` — 等待前置任务完成
- `🚧 in_progress` — 正在实现
- `✅ completed` — 已完成并通过验证
- `🔀 split → NNa, NNb` — 被拆分为子任务（原任务不再实现）
- `➕ discovered` — 实现过程中发现的新任务（编号在末位，不插入中间）

### Step 3: 任务实现循环

对每个可执行的任务（状态=`pending` 且前置依赖已满足）：

#### 3a. 展示任务

读取 `docs/features/<feature-name>/tasks/<NN>-<task-name>.md`，向用户展示：

> ---
> ## 📋 任务 01/07：Project scaffold
>
> **描述**：创建最小化项目骨架...
>
> **验收标准**：
> - [ ] package.json 存在
> - [ ] .gitignore 存在
>
> **涉及文件**：package.json, .gitignore
>
> 开始实现？
> ---

等待用户确认（或自动继续，如果用户之前已表示"全部自动执行"）。

#### 3b. 更新状态为 🚧 in_progress

修改 `tasks/README.md` 中对应行的状态从 `⏳ pending` 改为 `🚧 in_progress`。

#### 3c. 实现代码

按照任务描述和验收标准编写代码。遵循以下原则：

- **最小实现** — 只写任务要求的内容，不额外添加
- **引用需求** — 如有疑问，回看 `REQUIREMENT.md`
- **遵循项目规范** — 检查 `CLAUDE.md` 中的约定

#### 3c-bis: Code Quality Gate（代码质量门禁）

实现代码后、任务验证前，运行项目代码质量检查：

1. **自动修复代码风格** — 执行 `CLAUDE.md` 的 `## Code Quality` 章节中定义的格式化命令
2. **静态分析检查** — 执行 `CLAUDE.md` 的 `## Code Quality` 章节中定义的静态分析命令
3. 格式化修复产生的 diff 自动纳入当前变更
4. 静态分析失败 → 进入 3f 修复循环（计入 3 轮上限），每轮修复后重新运行：格式化 → 静态分析 → 任务验证
5. 两项检查全部通过后，方可进入 3e 任务验证

**注意**：质量命令以 `CLAUDE.md` 中的定义为准。如项目无 `CLAUDE.md` 或其中无 `## Code Quality` 章节，跳过此步骤。

#### 3d. 运行时任务调整

实现过程中，可能会发现原计划的粒度或覆盖范围有问题。允许两种调整操作：

##### 拆分（任务太大）

当任务明显超出 1-2 小时的工作量时，将当前任务拆分为多个子任务：

1. 在 `tasks/` 下创建子任务文件：
   - `04a-<子任务-a>.md`
   - `04b-<子任务-b>.md`
   - 子任务文件格式与普通任务文件相同
2. 修改 `tasks/README.md`：
   - 原始任务 04 的状态改为 `🔀 split → 04a, 04b`
   - 在 04 行下方插入 04a、04b 行，状态 `⏳ pending`
   - 依赖链：04a 继承 04 的前置依赖，04b 依赖 04a，原依赖 04 的任务改为依赖 04b
3. 创建对应 `artifacts/04a-*.log` 和 `artifacts/04b-*.log`（空文件，后续填充）
4. 提交拆分 commit：
   ```
   refactor(<feature-name>): split task 04 into 04a/04b
   ```
5. 从 04a 开始继续实现

##### 追加（发现遗漏）

当实现过程中发现遗漏——某个必要的步骤不在原计划中：

1. 在 `tasks/` 下创建新任务文件，编号为当前最大编号 +1，如 `08-<task-name>.md`
2. 修改 `tasks/README.md`：
   - 在表格末尾追加一行，状态 `➕ discovered`，依赖列填具体的已完成任务编号
   - 更新总任务数
3. 提交追加 commit：
   ```
   feat(<feature-name>): append discovered task 08 - <task-name>
   ```
4. 不中断当前任务，继续当前工作

**约束**：
- 只能拆分当前正在做的任务（`🚧 in_progress`）
- 只能追加新任务到列表末尾（编号递增，不插入中间）
- 原计划中 feature-analyzer 产出的任务编号和行永远不改动
- 每次调整单独 commit

#### 3e. 验证并持久化产物

1. 确保产物目录存在：`mkdir -p docs/features/<feature-name>/tasks/artifacts`
2. 从任务文件中的「验证方式」区域提取 bash 命令
3. **只执行任务定义中列出的命令**，不自行添加额外验证步骤——额外步骤未经审核，可能用错命令产生误导
4. **逐条检查退出码**：将命令写入临时脚本，`set -e` 开头，每条命令前加 `echo` 分隔符标识步骤；任一条命令失败（非零退出码）即整体失败
5. 将验证命令输出**同时**打印到终端并写入产物文件：

```bash
# 将任务验证命令写入临时脚本（set -e 确保任一步失败即中止）
cat > /tmp/verify-task-<NN>.sh << 'VERIFY_EOF'
#!/bin/bash
set -e
# ... 任务「验证方式」中的命令，每条前加 echo 分隔符 ...
VERIFY_EOF
bash /tmp/verify-task-<NN>.sh 2>&1 | tee docs/features/<feature-name>/tasks/artifacts/<NN>-<task-name>.log
VERIFY_EXIT=$?
```

产物文件头部需包含元信息（在运行前写入）：

```bash
ARTIFACT="docs/features/<feature-name>/tasks/artifacts/<NN>-<task-name>.log"
TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')
printf "# 任务 <NN>: <任务标题>\n# 验证时间: %s\n" "$TIMESTAMP" > "$ARTIFACT"
```

验证结果判断（基于 `$VERIFY_EXIT` 退出码，**不是**人工读日志判断）：
- `VERIFY_EXIT` = 0 → 在产物文件末尾追加 `✅ PASSED` 标记，进入步骤 3g
- `VERIFY_EXIT` ≠ 0 → 在产物文件末尾追加 `❌ FAILED (exit=$VERIFY_EXIT)` 标记，进入步骤 3f 修复循环

**硬性规则**：日志中如有明显的 `[critical]`、`Error thrown`、`exception`、`stack trace` 等字样，即使退出码意外为 0 也视为失败。禁止用"功能实际可工作"为由跳过验证失败。

#### 3f. 修复循环（最多 3 轮）

```
验证失败（产物文件末尾已有 ❌ FAILED）
    ↓
分析 artifacts/<NN>-*.log 中的失败输出 → 定位问题根因
    ↓
修改代码 → 重新验证（覆盖写入产物文件）
    ↓
仍失败且 < 3 轮 → 回到分析
    ↓
3 轮仍未通过 → 暂停，产物文件保留最后一轮失败日志
```

- 每轮修复聚焦于失败输出的具体信息（含静态分析错误、任务验收失败）
- 每轮修复后按顺序重新运行：格式化 → 静态分析 → 任务验证，确认全部通过
- 不要猜测 — 根据 `artifacts/<NN>-*.log` 中的错误信息精准修改
- 每轮重新验证时覆盖写入产物文件（保留最新一轮结果）
- 3 轮后仍未通过，向用户清晰说明：
  - 哪个验收标准失败
  - 已尝试的修复方式
  - 产物文件路径（供用户查看完整输出）
  - 建议的下一步

#### 3g. 标记完成

修改 `tasks/README.md`：
1. 将对应行的状态从 `🚧 in_progress` 改为 `✅ completed`
2. 在证据列填入 `[查看](artifacts/<NN>-<name>.log)`
3. 更新进度统计（已完成数 + 进度百分比）

#### 3h. 单任务 commit

每完成一个任务就提交一次 commit：

```bash
git add -A
git commit -m "$(cat <<'EOF'
feat(<feature-name>): <task-title> (task <NN>/<total>)

<task description summary>

Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>
EOF
)"
```

Commit 消息格式：
- type 使用 `feat`（新功能）或 `fix`（修复）
- scope 使用 feature-name
- body 包含任务编号和简要描述

### Step 4: 全部任务完成后的集成验证

当 `tasks/README.md` 中所有任务处于终态（`✅ completed` 或 `🔀 split`）时：

```bash
bash docs/features/<feature-name>/verify.sh 2>&1 | tee docs/features/<feature-name>/tasks/artifacts/integration-verify.log
```

- ✅ 集成验证通过 → 在产物文件末尾追加 `✅ INTEGRATION PASSED`，进入 Step 5
- ❌ 集成验证失败 → 诊断并修复，重新验证（覆盖写入产物文件）

### Step 5: 最终提交

```bash
git add -A
git commit -m "$(cat <<'EOF'
feat(<feature-name>): complete implementation

All N tasks completed and verified.

Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>
EOF
)"
```

向用户报告：

> ✅ **<feature-name> 实现完成！**
>
> - 已完成 N/N 个任务
> - 集成验证通过
> - 验证产物保存在 `docs/features/<feature-name>/tasks/artifacts/`
> - 共提交 N+1 个 commit
>
> 运行 `git log --oneline` 查看提交历史。运行 `cat docs/features/<feature-name>/tasks/artifacts/*.log` 查看所有验证证据。

## 🧠 Reasoning Strategy

1. **先读后写** — 实现每个任务前，先阅读涉及文件的当前状态
2. **验证驱动** — 验收标准和验证命令是唯一真相来源，代码服务于通过验证
3. **精准修复** — 验证失败时基于错误信息修复，不要猜测或重构无关代码
4. **粒度假定** — 每个任务应能在 1-2 小时内完成；如果需要更长时间，考虑建议拆分
5. **依赖感知** — 后续任务的实现基于前置任务的代码产出，保持代码上下文连续

---

## 🚫 Constraints

**禁止**：

- 跳过验证直接标记任务完成
- 验证后不将输出持久化到 `artifacts/` 目录
- 在验证失败时忽略错误继续下一个任务
- 自行在任务定义的「验证方式」之外添加额外验证步骤
- 验证日志中有明显 `[critical]`/`Error`/`exception` 时仍标记通过
- 修改或删除 feature-analyzer 产出的原计划任务行（任务可拆分可追加，但原编号和条目不可变）
- 同时实现多个任务（任务必须按依赖顺序逐一完成）
- 修改任务文件中的验收标准以适应实现
- 实现任务描述范围之外的功能
- 在验证不通过时 commit
- 在代码质量检查（格式化 / 静态分析）未通过时进入 3g 标记任务完成
- 跳过或忽略 `CLAUDE.md` 中定义的代码质量检查步骤
- 拆分非当前 `🚧 in_progress` 的任务

## 🔧 Edge Cases

| 场景 | 处理方式 |
|------|----------|
| 用户中断后恢复 | 读取 `README.md`，从第一个 `⏳ pending` 任务继续 |
| 任务无前置依赖 | 可直接开始（如 task 01） |
| 前置任务未完成 | 跳过，先完成前置任务 |
| 验证命令为空 | 提醒用户补充验证方式，根据验收标准自行验证 |
| README.md 格式不标准 | 先修复 README.md 格式再继续 |
| 已全部完成 | 报告"所有任务已完成"，询问是否需要重新验证 |
| artifacts/ 目录不存在 | `mkdir -p` 自动创建 |
| 任务太大需拆分 | 拆分为子任务，原任务标记 `🔀 split`，单独 commit |
| 发现遗漏任务 | 追加到表格末尾，标记 `➕ discovered`，单独 commit |
| 拆分后再恢复 | 读 README，跳过 `🔀 split`，取下一个 `⏳ pending` |
| 项目无 CLAUDE.md | 跳过 Code Quality Gate，仅执行任务自定义验证 |
| 静态分析报错无法修复 | 3 轮后暂停，产物文件中记录错误和建议下一步 |
| 格式化导致意外变更 | 检查 diff，如格式化规则有问题，修正 CLAUDE.md 中的规则定义 |

- 验证结果直接展示给用户（PASS/FAIL 计数）
